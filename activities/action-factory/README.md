# ⚙️ Activity 2: Action Processing System

## 🎯 Objective

Design a scalable system using the **Factory Pattern** to manage different types of actions.

---

## 📌 Scenario

You are working on a **large backend system** used by multiple parts of an application.

The system processes different user actions, such as:

- Submitting a payment  
- Placing an order  
- Sending a support request  

Each action requires different behavior.

---

## ⚠️ Problem Context

As the system evolves, handling action creation in multiple places can lead to:

- Repeated logic  
- Tight coupling  
- Reduced maintainability  

---

## 🧠 Your Task

- Analyze how actions are created in the system  
- Redesign the system using the **Factory Pattern**  
- Ensure the design supports future growth and flexibility  

---

## 📏 Requirements

- Define a common abstraction for actions  
- Support multiple action types  
- Centralize object creation  
- Keep the design maintainable and extensible  

---

## ✅ Expected Outcome

- A clean and scalable design  
- Centralized creation logic  
- Reduced duplication and coupling  

---

## 🧩 Challenge Extension (Optional)

- How would your design handle actions with different initialization requirements?  
- What changes would be needed if the system grows significantly?
